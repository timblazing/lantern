## Why these four are written first

Plans 001–004 are the **backend/data foundation** and follow CONTEXT.md's
recommended path (line 53): "React/Vite UI plus Bun/Hono API first, demo scanner
second, SQLite persistence third." They are written in full now because their
shape is well-determined by the existing scaffold and the committed decisions.

Plans 005–007 are deliberately **not written yet**: their detail depends on the
exact contract and route shapes that 002/003 lock in (e.g. the typed client and
table columns key off the final `Host` schema; the config screen keys off the
final `appConfig` schema). Writing them now would specify against an unsettled
contract and produce drift. Expand them after 002–003 land (or sooner if you want
to fix the contract first — say so and they can be drafted against the plan-as-written).

### Summaries of the not-yet-written plans

- **005 — Typed API client + host table UI.** In `apps/web`: a small fetch
  wrapper that parses responses with the `@lantern/shared` schemas (runtime-safe
  typing), TanStack Table for the host list (search, column filters, sort, row
  selection, bulk delete), a host detail drawer (shadcn `Sheet`/`Dialog`) showing
  history, and a manual "Rescan" button hitting `POST /api/scan`. Demo mode is
  automatic — the server already returns demo data. Components come from the
  shadcn registry already configured (`components.json`, style `radix-mira`).
- **006 — Config screens + action flows.** shadcn `Tabs`/`Card`/`Field` forms for
  scan/storage/notification config (validated with the shared `appConfig` schema),
  plus the per-host action UI: Wake-on-LAN, port check, and a notification test
  button. Requires server endpoints for reading/writing config (added in this
  plan's server portion) on top of the action routes stubbed in Plan 003.
- **007 — Linux `arp-scan` adapter + packaging.** An `ArpScanScanner`
  implementing the `Scanner` interface from Plan 003 by shelling out to
  `arp-scan` (parsing MAC/IP/vendor), selected via config when `interfaces` is
  non-empty (else demo). Plus a multi-stage Dockerfile (build web, run Bun server,
  serve the built UI), a mounted `config.yaml` loader with env overrides, and the
  `data/` volume for SQLite. Linux Docker only (CONTEXT line 44).

## Dependency notes

- **001 first, always.** The Vite app currently lives at the repo root; 002–007
  all assume the `apps/web` / `apps/server` / `packages/shared` layout 001 creates.
- **002 before 003 and 005.** Both the server and the UI import domain/contract
  types from `@lantern/shared`; the schemas must exist first.
- **003 before 004.** 004 swaps `InMemoryHostStore` for a SQLite store behind the
  **same `HostStore` interface** 003 defines — that interface must exist first.
- **003 before 005.** The UI needs a running API to call.
- **004 before 007.** Real scans should persist history across restarts; the
  scheduled-scan loop and retention land on top of SQLite, not the in-memory store.
- **005 before 006.** Config/action UI builds on the client + table shell.

- Package manager / runner: **Bun 1.3.14** (confirmed on PATH). Never use npm/yarn/pnpm.
- Test runner: **`bun test`** (built-in). The repo has **no tests yet** — Plan 002

## Deferred follow-ups surfaced while planning (not yet plans)

These were noted in plan "Maintenance notes" and are worth tracking:

- **Scheduled scanning loop** (run `applyScan` every `scanConfig.intervalSeconds`)
  — deferred until SQLite exists (Plan 004) so history survives restarts.
- **History retention/trimming** (`storageConfig.historyRetentionDays`) — periodic
  delete of old `host_history` rows; not in Plan 004.
- **Real Wake-on-LAN and TCP port-check** — Plan 003 ships demo stubs; real
  implementations want the scanner/packaging context of Plan 007.
- **Static serving of the web build from the server** — Plan 003 leaves a stub;
  finalized in Plan 007's Dockerfile (single-container serving).
- **Auth middleware** — explicitly out of milestone scope (CONTEXT line 41);
  routes are mounted under `/api/*` to leave room for it.
- **Pre-existing lint error in `button.tsx`** — `react-refresh/only-export-components`
  at `apps/web/src/components/ui/button.tsx:65` (`export { Button, buttonVariants }`).
  Verified present since the initial commit `969f681`; **not** introduced by Plan
  001's restructure. Standard shadcn fix: move `buttonVariants` into a separate
  module (e.g. `button-variants.ts`) so the component file only exports the
  component. Fold this into Plan 005 (host-table UI), or do it as a one-line
  cleanup. Until then `bun run --cwd apps/web lint` exits 1 on this one rule.

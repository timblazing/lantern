## Why these four are written first

Plans 001‚Äì004 are the **backend/data foundation** and follow CONTEXT.md's
recommended path (line 53): "React/Vite UI plus Bun/Hono API first, demo scanner
second, SQLite persistence third." They are written in full now because their
shape is well-determined by the existing scaffold and the committed decisions.

Plans 005‚Äì007 are deliberately **not written yet**: their detail depends on the
exact contract and route shapes that 002/003 lock in (e.g. the typed client and
table columns key off the final `Host` schema; the config screen keys off the
final `appConfig` schema). Writing them now would specify against an unsettled
contract and produce drift. Expand them after 002‚Äì003 land (or sooner if you want
to fix the contract first ‚Äî say so and they can be drafted against the plan-as-written).

### Summaries of the not-yet-written plans

- **005 ‚Äî Typed API client + host table UI.** In `apps/web`: a small fetch
  wrapper that parses responses with the `@lantern/shared` schemas (runtime-safe
  typing), TanStack Table for the host list (search, column filters, sort, row
  selection, bulk delete), a host detail drawer (shadcn `Sheet`/`Dialog`) showing
  history, and a manual "Rescan" button hitting `POST /api/scan`. Demo mode is
  automatic ‚Äî the server already returns demo data. Components come from the
  shadcn registry already configured (`components.json`, style `radix-mira`).
- **006 ‚Äî Config screens + action flows.** shadcn `Tabs`/`Card`/`Field` forms for
  scan/storage/notification config (validated with the shared `appConfig` schema),
  plus the per-host action UI: Wake-on-LAN, port check, and a notification test
  button. Requires server endpoints for reading/writing config (added in this
  plan's server portion) on top of the action routes stubbed in Plan 003.
- **007 ‚Äî Linux `arp-scan` adapter + packaging.** An `ArpScanScanner`
  implementing the `Scanner` interface from Plan 003 by shelling out to
  `arp-scan` (parsing MAC/IP/vendor), selected via config when `interfaces` is
  non-empty (else demo). Plus a multi-stage Dockerfile (build web, run Bun server,
  serve the built UI), a mounted `config.yaml` loader with env overrides, and the
  `data/` volume for SQLite. Linux Docker only (CONTEXT line 44).
